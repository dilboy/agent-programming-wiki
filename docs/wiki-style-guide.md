# Wiki 编辑风格指南

本指南基于 P0 批次审核经验总结，供所有贡献者遵循，确保内容一致性、准确性和可维护性。

---

## 1. Frontmatter 规范

所有内容页面（除 `index.md` 外）必须包含以下 YAML frontmatter：

```yaml
---
title: 页面标题
description: 一句话描述（50 字以内）
tags: [标签1, 标签2]
category: XX-分类名称
difficulty: 初级|中级|高级
created: YYYY-MM-DD
updated: YYYY-MM-DD
---
```

### 1.1 必填字段

| 字段 | 规则 |
|------|------|
| `title` | 与文件名一致（不含 `.md`），H1 标题必须与 `title` 完全一致 |
| `description` | 一句话概括，不超过 50 字，用于 SEO 和摘要 |
| `tags` | 3-5 个，中文优先，关键术语可保留英文 |
| `category` | **必须带数字前缀**，如 `01-基础概念`、`02-架构模式`。不允许简写为 `基础概念` |
| `difficulty` | 仅允许 `初级` / `中级` / `高级` |
| `created` / `updated` | ISO 日期格式 `YYYY-MM-DD`；修改内容后必须更新 `updated` |

### 1.2 一致性检查清单

- [ ] `category` 与文件所在目录一致
- [ ] H1 (`# 标题`) 与 `title` 完全一致
- [ ] 无重复键、无 YAML 语法错误

## 2. 术语与命名规范

### 2.1 中英混排间距

- **英文与中文之间必须加空格**：`LLM 推理` ✅，`LLM推理` ❌
- **英文标点与中文之间不加空格**：`Agent（智能体）` ✅，`Agent （智能体）` ❌
- **数学运算符前后加空格**：`Token 消耗 × 轮数` ✅，`Token 消耗×轮数` ❌
- **代码块内、URL、文件名除外**

### 2.2 术语统一

| 规范写法 | 避免使用 |
|----------|----------|
| Agent | agent、智能体（首次出现可标注） |
| LLM | llm、大模型（首次出现可标注） |
| Function Calling | function calling、函数调用（视语境） |
| ReAct | ReAct、React、react |
| 提示链 | Prompt Chaining（可并用） |

### 2.3 能力层级 vs 自主性分级

**这是两个不同维度，不可混淆：**

- **能力层级**（L1-L5）：衡量 Agent 能完成的任务复杂度，见 `Agent-能力模型.md`
  - L1 指令遵循 → L2 工具调用 → L3 多步推理 → L4 自主规划 → L5 持续学习
- **自主性分级**（L0-L4）：衡量 Agent 的自主决策程度，见 `什么是Agent.md`
  - L0 无自主 → L1 工具自主 → L2 步序自主 → L3 目标自主 → L4 演进自主

**引用时必须明确区分，不可写"参考 Agent 能力模型"来支撑自主性分级。**

## 3. Markdown 格式规范

### 3.1 标题层级

- 每页必须有且仅有一个 H1 (`#`)
- H2 (`##`) 用于主要章节，H3 (`###`) 用于子章节
- 不允许跳级（如 H2 后直接 H4）
- 标题末尾不加标点

### 3.2 代码块

- 必须标注语言：` ```python `、` ```typescript `、` ```mermaid `
- 纯文本/对话示例标注为 ` ```text `
- 禁止无语言标签的代码块

### 3.3 表格

- 必须包含表头分隔行 `|------|------|`
- 列数在所有行保持一致
- 单元格内换行使用 `<br/>`

### 3.4 引用与来源

- 论文引用格式：`作者（年份）在《论文标题》中提出……`
- arXiv 链接直接给出：`[ReAct](https://arxiv.org/abs/2210.03629)`
- 书籍引用格式：`Russell, S. J., & Norvig, P. (2020). *Artificial Intelligence: A Modern Approach* (4th ed.). Pearson.`

## 4. Wiki 链接规范

### 4.1 语法

- 内部链接：`[[页面标题]]` 或 `[[页面标题|显示文本]]`
- **使用页面标题（文件名，不含 `.md`），不使用目录前缀**：`[[什么是Agent]]` ✅，`[[01-基础概念/什么是Agent]]` ❌
- 外部链接：标准 Markdown `[text](url)`

### 4.2 验证

- 所有 Wiki 链接必须在构建前验证可解析
- 新增页面时，同步更新相关页面的 "延伸阅读" 章节

---

## 5. Mermaid 图表规范

- 优先使用 `graph TD`（从上到下）或 `graph LR`（从左到右）
- 时序图使用 `sequenceDiagram`
- 节点标签内的换行使用 `<br/>`
- 图表后必须跟一段文字说明，不可孤立存在

---

## 6. 语气与风格

- **专业、权威、有帮助**，禁止营销腔
- 使用主动语态和短句，便于扫描阅读
- 首次出现的专业术语必须给出简明解释
- 技术声明必须有来源或标注为"业界共识"
- 避免绝对化表述（如"最好的"、"唯一的"），改用"在生产环境中常见"、"当前主流做法"

---

## 7. 审核流程

1. **自测**：提交前运行 `npm run docs:build`，确保构建通过
2. **清单检查**：对照本指南逐条自检
3. **编辑审核**：ContentEditor 对照 7 点 checklist 审核
4. **技术复核**：涉及架构决策或性能数据时，标记 CTO 复核

### 7.1 退回标准

以下情况直接退回 TechnicalWriter 重写：

- 核心概念定义与已有文章矛盾
- 代码示例存在明显语法错误或无法运行
- 技术声明无来源且无法验证
- 大量术语前后不一致

以下情况由 ContentEditor 直接修改：

- Frontmatter 字段缺失或格式错误
- 中英混排间距问题
- Wiki 链接格式错误或死链
- 标题层级问题
- 错别字和标点问题
