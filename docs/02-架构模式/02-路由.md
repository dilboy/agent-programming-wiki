# 路由（Routing）

## 定义

**路由（Routing）** 是根据输入的特征（内容、类型、意图等），将请求分发到最合适的处理路径或专用模型的模式。

```mermaid
graph TD
    A[用户输入] --> B[分类器/路由器]
    B -->|类型A| C[处理路径A]
    B -->|类型B| D[处理路径B]
    B -->|类型C| E[处理路径C]
    C --> F[输出]
    D --> F
    E --> F
```

## 适用场景

- 系统需要处理多种不同类型的任务
- 不同任务类型需要不同的处理方式或专用模型
- 需要按领域/复杂度/敏感度分流
- 希望优化成本和性能（简单任务用轻量模型）

## 典型示例：客服系统

```mermaid
graph TD
    A[用户问题] --> B[意图分类]
    B -->|订单查询| C[订单处理模块]
    B -->|技术支持| D[技术支持模块]
    B -->|投诉建议| E[投诉处理模块]
    B -->|闲聊| F[闲聊回复模块]
    C --> G[回复用户]
    D --> G
    E --> G
    F --> G
```

## 代码示例

### 基于 LLM 的分类路由

```python
from enum import Enum

class QueryType(Enum):
    ORDER = "订单查询"
    TECH = "技术支持"
    COMPLAINT = "投诉建议"
    CHAT = "闲聊"

def classify_query(user_input: str) -> QueryType:
    """使用 LLM 对输入进行分类"""
    prompt = f"""请将用户输入分类为以下类型之一：
- 订单查询
- 技术支持
- 投诉建议
- 闲聊

用户输入：{user_input}

只返回类型名称，不要其他解释。"""
    
    result = llm.invoke(prompt).strip()
    return QueryType(result)

def route_and_process(user_input: str) -> str:
    query_type = classify_query(user_input)
    
    handlers = {
        QueryType.ORDER: handle_order,
        QueryType.TECH: handle_tech,
        QueryType.COMPLAINT: handle_complaint,
        QueryType.CHAT: handle_chat,
    }
    
    handler = handlers[query_type]
    return handler(user_input)
```

## 优缺点

| 优点 | 缺点 |
|------|------|
| 各路径可独立优化 | 分类错误会导致整个请求处理失败 |
| 可为不同任务选择最适合的模型 | 增加一层延迟（分类步骤） |
| 便于扩展新类型 | 分类边界模糊时效果下降 |
| 成本优化（简单任务用轻量模型） | 需要维护分类器和多条路径 |

## 反模式与修复

| 反模式 | 问题描述 | 影响 | 修复方案 |
|--------|----------|------|----------|
| 分类器过于复杂 | 使用大型 LLM 或复杂 Agent 做路由分类本身，分类步骤延迟和成本超过实际处理 | 路由层变成瓶颈：分类耗时 3-5s，实际处理只需 1s；成本中分类占比过高 | 分类器应使用轻量模型或少量 few-shot prompt，优先考虑基于规则/关键词的快速分类，仅在边界模糊时使用 LLM |
| 缺少兜底路径 | 路由只覆盖已知类型，用户输入落在所有分类之外时无处可去 | 未分类请求直接报错或被丢弃，用户得到无意义的错误消息；线上事故频发 | 始终设置 `default` 兜底路径，将未分类请求路由到通用处理模块或人工队列 |
| 分类边界重叠 | 多个分类的定义存在语义交叉（如"技术支持"与"产品咨询"），未明确边界 | 同类请求被随机分发到不同路径，处理逻辑和回复风格不一致；用户反馈质量波动大 | 在分类定义中加入正例和反例，使用 few-shot 示例明确边界；对置信度低的请求走合并处理或人工判定 |
| 路由决策不可观测 | 路由器只输出分类结果，不记录分类置信度和推理依据 | 出现分类错误时无法定位原因，难以迭代优化分类器；无法统计各路径的流量分布 | 在路由结果中记录 `category`、`confidence`、`reasoning`，接入日志和监控系统，定期分析分类质量 |
| 静态路由无法演化 | 路由规则硬编码在代码中，新增业务类型需要修改代码并重新部署 | 新业务上线时路由无法及时更新，所有请求被错误分类到旧类型；迭代周期长 | 将路由规则配置化（配置文件或数据库），支持运行时热更新；参考 [[04-编排器-工作者]] 的动态任务分配思路 |
| 分类与处理耦合 | 分类器的 prompt 中包含了具体处理逻辑的描述，导致路由和处理路径强耦合 | 修改某条处理路径的逻辑时需要同步修改分类器 prompt，维护成本高，容易引入不一致 | 分类器只做类型判断（如返回 enum 值），处理逻辑完全封装在各自的 handler 中，实现关注点分离 |

**分类器过于复杂**是路由模式中最常见的反模式。许多开发者为了追求分类准确率，直接使用 GPT-4 级别的强模型做意图分类，但分类本身是一个相对简单的任务——通常只需要识别几个固定类别。如果分类器的延迟和成本超过了实际处理路径，那路由带来的优化效果就被完全抵消。建议先尝试基于关键词和正则的规则引擎，覆盖 80% 的明确场景，仅对剩余 20% 的模糊输入使用轻量 LLM 分类。

**缺少兜底路径**同样是高风险反模式。现实世界中的用户输入远比预设分类丰富，总会有不属于任何已定义类别的情况。如果没有兜底策略，这些请求会直接失败。一个健壮的路由系统应该在任何情况下都能给出有意义的响应——即使只是"我无法确定您的需求，已为您转接人工客服"。

## 权衡分析

路由模式的核心设计选择是**如何分类请求以及分几条路径**。

### 分类精度 vs 延迟开销

| 维度 | LLM 分类 | 规则引擎 | 嵌入向量相似度 |
|------|----------|----------|---------------|
| 准确率 | 高（理解语义） | 中（关键词匹配） | 中高（语义相近） |
| 延迟 | 高（一次 LLM 调用） | 极低（微秒级） | 低（向量计算） |
| 成本 | 高（Token 消耗） | 无 | 低（一次嵌入） |
| 维护成本 | 低（改 Prompt） | 高（规则膨胀） | 中（需标注数据） |
| 适用场景 | 类别多、边界模糊 | 类别少、规则明确 | 有历史数据可训练 |

### 路由粒度的权衡

- **粗粒度路由（2-4 条路径）**：分类器准确率高，但每条路径需要处理的任务类型多，内部逻辑复杂
- **细粒度路由（10+ 条路径）**：每条路径高度专业化，但分类器准确率下降，类别间边界模糊
- **级联路由**是折中方案：先粗分类（准确率高），再细分类（专业化），但增加一层延迟

### 成本优化的取舍

- **多模型路由**（简单任务用轻量模型）可节省 60-80% 成本，但需要可靠的复杂度评估器
- 复杂度评估器本身可能出错——错误地将复杂任务分配给轻量模型会导致质量严重下降
- **保守策略**：宁可用强模型处理简单任务，也不要用轻量模型处理复杂任务

### 何时选择路由

- 系统需要处理**3 种以上**明显不同的任务类型
- 不同任务需要**不同的处理策略或模型**
- 需要**按成本/质量**分级处理

### 何时避免路由

- 任务类型**少于 3 种**——直接用 if-else 或提示链更简单
- 分类边界**极其模糊**——考虑让 LLM 自主判断（ReAct 模式）
- 路径之间**需要共享上下文**——编排器-工作者模式更合适

## 最佳实践

1. **分类器要简单**：分类任务本身应该是简单的，避免"分类器也变成 Agent"
2. **兜底路径**： always 提供默认/兜底处理路径
3. **持续优化**：收集分类错误样本，迭代优化分类器
4. **置信度阈值**：当分类置信度低时，降级到通用处理或人工介入

## 与其他模式的关系

- **vs [[01-提示链|提示链]]**：路由是多分支选择，提示链是单线执行
- **vs [[03-并行化|并行化]]**：路由选择一条路径，并行化同时走多条路径
- **vs [[04-编排器-工作者|编排器-工作者]]**：路由是静态分类，编排器动态分解任务
