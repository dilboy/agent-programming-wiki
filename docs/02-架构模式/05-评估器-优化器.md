# 评估器-优化器（Evaluator-Optimizer）

## 定义

**评估器-优化器（Evaluator-Optimizer）** 是一种迭代优化模式：生成器（Generator）产出结果，评估器（Evaluator）检查质量并给出反馈，生成器根据反馈改进，循环直到满足质量标准。

```mermaid
graph TD
    A[输入] --> B[生成器<br/>生成初稿]
    B --> C[评估器<br/>质量评估]
    C -->|未达标| D[反馈]
    D --> B
    C -->|达标| E[最终输出]
```

## 适用场景

- 输出质量有明显可评估的标准
- 迭代改进可以持续提升质量
- 评估比生成更容易（或成本更低）
- 需要高质量输出的场景（代码、文档、翻译等）

## 典型示例：代码生成与修复

```mermaid
graph TD
    A[需求描述] --> B[生成代码]
    B --> C[运行测试]
    C -->|测试失败| D[错误分析]
    D --> B
    C -->|测试通过| E[代码审查]
    E -->|未通过| F[审查意见]
    F --> B
    E -->|通过| G[交付代码]
```

## 代码示例

### Python 实现

```python
class EvaluatorOptimizer:
    def __init__(self, generator_llm, evaluator_llm, max_iterations=5):
        self.generator = generator_llm
        self.evaluator = evaluator_llm
        self.max_iterations = max_iterations
    
    def generate(self, prompt: str, feedback: str = "") -> str:
        """生成内容，可选携带反馈"""
        full_prompt = prompt
        if feedback:
            full_prompt += f"\n\n之前的反馈：{feedback}"
        return self.generator.invoke(full_prompt)
    
    def evaluate(self, output: str, criteria: str) -> tuple[bool, str]:
        """评估输出，返回(是否通过, 反馈)"""
        eval_prompt = f"""评估以下输出是否满足标准。

标准：{criteria}

输出：
{output}

请判断：通过/未通过
改进建议："""
        
        result = self.evaluator.invoke(eval_prompt)
        passed = "通过" in result
        feedback = extract_feedback(result)
        return passed, feedback
    
    def run(self, prompt: str, criteria: str) -> str:
        """运行生成-评估循环"""
        output = self.generate(prompt)
        
        for i in range(self.max_iterations):
            passed, feedback = self.evaluate(output, criteria)
            if passed:
                return output
            
            output = self.generate(prompt, feedback)
        
        # 达到最大迭代次数，返回最佳结果
        return output
```

### 代码修复场景

```python
def generate_and_fix_code(requirement: str, test_cases: list) -> str:
    """生成代码并通过测试用例迭代修复"""
    
    code = llm.invoke(f"根据需求编写 Python 函数：\n{requirement}")
    
    for iteration in range(5):
        # 运行测试
        results = run_tests(code, test_cases)
        
        if all(r.passed for r in results):
            return code
        
        # 构建失败信息
        failures = "\n".join(
            f"测试：{r.test_name}\n错误：{r.error}"
            for r in results if not r.passed
        )
        
        # 修复代码
        fix_prompt = f"""修复以下代码以通过测试。

当前代码：
{code}

失败的测试：
{failures}

请修复代码，只返回修复后的完整代码。"""
        
        code = llm.invoke(fix_prompt)
    
    raise Exception("无法在最大迭代次数内修复代码")
```

## 优缺点

| 优点 | 缺点 |
|------|------|
| 持续提升输出质量 | 延迟增加（多轮迭代） |
| 评估和生成可分离优化 | 可能陷入局部最优 |
| 质量有明确标准 | 评估标准本身可能不完美 |
| 适合对质量敏感的场景 | 成本随迭代次数增加 |

## 反模式与修复

| 反模式 | 问题描述 | 影响 | 修复方案 |
|--------|----------|------|----------|
| 生成器与评估器是同一模型 | 用同一个 LLM（甚至同一个 prompt）既生成内容又评估质量，即"自评" | 模型倾向于认可自己的输出，评估结果虚高；系统陷入"自我肯定"循环，实际质量未提升但评估反复显示"通过" | 生成器和评估器必须使用不同模型（或至少不同 prompt/temperature）；评估器 prompt 应明确要求以批判视角审查 |
| 评估标准模糊不清 | 评估标准写成"确保质量好""写得专业"等主观描述，无量化指标 | 评估器每次对同一输出给出不同判断（通过/不通过随机切换），迭代循环无法收敛；浪费大量 LLM 调用成本 | 将评估标准拆解为可检查的具体规则（如"代码必须通过所有测试用例""翻译准确率 > 95%"），使用结构化评分（1-5 分制 + 阈值） |
| 无限迭代不收敛 | 设置了 `max_iterations` 但未监控迭代趋势，生成器在某些反馈下反复振荡而非收敛 | 达到最大迭代次数后返回的仍是低质量结果；每次运行消耗 N 倍 LLM 调用成本，用户体验极差 | 检测连续两次迭代的输出相似度，若变化量低于阈值则提前终止；记录每次评估分数，分数无改善时触发早停（early stopping） |
| 反馈信息不足或过于笼统 | 评估器只返回"未通过"，不给出具体的改进建议 | 生成器收到模糊反馈后只能随机调整方向，改进效率极低；可能在同一个问题上反复迭代 | 评估器 prompt 必须要求输出结构化反馈：具体问题位置、问题类型、改进建议；将反馈拆解为逐条可操作的改进项 |
| 过度优化导致过拟合 | 生成器针对评估器的特定偏好反复优化，产出在评估标准下得分极高但实际使用价值低 | 代码只通过了提供的测试用例但存在边界 bug；文章迎合评分规则但可读性下降，实际用户满意度低 | 评估维度多元化（准确性 + 可读性 + 安全性），使用不同评估器分别打分后加权汇总；引入人工抽检环节验证评估器本身的质量 |
| 评估成本超过生成成本 | 每轮迭代中评估器调用复杂度高于生成器（如评估器要求多维度打分 + 详细反馈），单次评估比生成更贵 | 迭代 5 次的总成本中评估占 70%+；投入产出比不合理，不如直接使用更强的生成器一次性完成 | 评估分层：先用规则检查（正则、测试用例）快速过滤明显问题，通过后再用 LLM 做精细评估；简单任务只做规则评估 |

**生成器与评估器使用同一模型**是评估器-优化器模式中最根本的设计缺陷。LLM 存在明确的自我偏好（self-bias）——研究表明，模型对自己生成的内容评分显著高于对人类内容的评分。当生成器和评估器是同一个模型时，评估器几乎不会否定生成器的输出，整个迭代循环形同虚设。最可靠的修复是使用不同厂商的模型（如生成用 Claude，评估用 GPT-4），或者至少使用截然不同的 system prompt 和 temperature 设置。

**评估标准模糊**直接决定了迭代循环能否收敛。如果标准是"文章要专业"，评估器可能第一轮说"通过"，第二轮又说"未通过"——因为它对"专业"的理解每次都在变化。将标准量化为可检查的规则（如"包含至少 3 个数据引用""段落间有逻辑过渡句"）后，评估结果会变得确定且可重复，迭代才能真正推动质量提升。

## 权衡分析

评估器-优化器的核心设计选择是**迭代深度 vs 成本/延迟、评估严格度 vs 收敛速度**。

### 迭代次数 vs 成本与质量

| 迭代次数 | 成本倍数 | 延迟倍数 | 质量提升 | 适用场景 |
|----------|----------|----------|----------|----------|
| 1 轮 | 2×（生成 + 评估） | 2× | 基础 | 低价值任务，快速原型 |
| 3 轮 | 4-6× | 4-6× | 显著 | 大多数生产场景 |
| 5 轮 | 6-10× | 6-10× | 边际递减 | 高价值任务（代码、翻译） |
| 10+ 轮 | >15× | >15× | 几乎无提升 | 通常不推荐 |

关键观察：**质量提升在 3-5 轮后趋于平缓**，但成本和延迟线性增长。大多数场景下 3 轮是性价比最优的迭代次数。

### 评估严格度的取舍

- **严格评估**：通过率低，迭代次数多，最终质量高，但成本和延迟也高
- **宽松评估**：通过率高，迭代次数少，成本低，但可能放过低质量输出
- **分级评估**是折中方案：先用宽松标准快速筛选（低成本），再用严格标准精细评估（高质量）

### 生成器与评估器的模型选择

- **同一模型**：成本最低，但存在自评估偏差——模型倾向于忽略自身常见的错误模式
- **不同模型**：可以互补盲区，但增加集成复杂度和成本
- **评估器用弱模型**：可降低评估成本，但可能无法准确判断高质量输出
- **评估器用强模型**：判断最准确，但评估成本可能超过生成成本

### 何时选择评估器-优化器

- 输出质量有**明确可衡量的标准**（如测试通过率、格式合规性）
- **评估比生成容易**（如运行测试比写代码简单）
- 任务对**质量敏感**（如生产代码、正式文档、客户回复）
- 有**明确的反馈信号**可以指导改进

### 何时避免评估器-优化器

- **没有清晰的评估标准**——主观质量（如"写得好不好"）难以驱动有效迭代
- 任务对**延迟极其敏感**——每轮迭代都是额外延迟
- **评估成本高于生成成本**——如需要人工评估，规模化不经济
- 生成质量**已经足够好**——迭代的边际收益不值得额外开销

## 最佳实践

1. **评估标准要明确**：模糊的评估标准会导致迭代无效
2. **设置迭代上限**：防止无限循环，设置 max_iterations
3. **评估器要独立**：避免生成器和评估器是同一个模型（自评估偏差）
4. **渐进式评估**：先快速粗筛，再精细评估
5. **记录迭代历史**：便于分析改进路径和调试
